# Гайд з авторингу контенту (Integral Trainer)

Правила, як додавати методи, задачі, кроки, підказки й формули, щоб усе тримало
**найвищу планку навчання**: просто, конкретно, без тупиків і без «магії».

> Золоте правило: студент 1 курсу, який «не дуже шарить», має зрозуміти крок
> **без сторонньої допомоги**. Якщо для розуміння треба щось, чого нема на екрані —
> цього бракує в контенті.

---

## 0. Принципи (чому саме так)

1. **Один крок = одна думка.** Не зливай два перетворення в один крок.
2. **Показуй, не згадуй.** Не «застосуємо правило» — а покажи, що саме і чому.
3. **Від абстрактного до конкретного.** Після формули з `x` завжди має бути шлях
   до прикладу на реальних числах (через `help.numeric`).
4. **Жодного тупика.** Якщо пояснення може не зайти — дай `help.simpler` (ще простіший рівень).
5. **Колір несе зміст** і тягнеться наскрізь (див. §6). Колір ≠ прикраса.
6. **Перевіряй математику похідною.** Похідна відповіді мусить дорівнювати підінтегральній функції.
7. **Мова:** українська, дружня, коротка. Жаргон — лише якщо поряд є підказка з поясненням.

## 2. Як додати НОВИЙ МЕТОД

У `src/lib/methods.ts` додай об'єкт у масив `methods`:

```ts
{
  id: "substitution",                  // kebab-case, унікальний; ним тегаємо кроки
  name: "Заміна змінної",              // повна назва
  short: "Складну функцію всередині заміняємо на t",  // суть однією фразою
  when: "Коли всередині є «функція у функції», напр. (3x+1)⁵.", // КОЛИ застосовувати
  description: "...",                   // повний, але простий опис
  example: "\\int (3x+1)^5\\,dx,\\quad t = 3x+1", // latex-приклад
  color: "#f472b6",                    // свій унікальний колір (див. §6)
}
```

Правила:
- **Не плоди дублів.** Степеневе правило — це частина `table`, а не окремий метод.
  Винесення множника — частина `linearity`. Додавай метод, лише якщо це справді
  окремий *прийом*.
- `color` — обери відмінний від уже зайнятих (нижче). Бейдж і картка беруть колір звідси.
- Картка в теорії та tooltip-бейджа з'являються **автоматично** — нічого більше не треба.

Зайняті кольори методів: `linearity` `#818cf8`, `table` `#34d399`,
`rewrite` `#fbbf24`, `substitution` `#f472b6`, `by-parts` `#a78bfa`, `verify` `#2dd4bf`.

## 3. Як додати НОВУ ЗАДАЧУ

У `src/lib/problems.ts` додай об'єкт у масив `problems`:

```ts
{
  id: "sqrt-x",                 // kebab-case, унікальний → це URL /problem/<id>
  title: "Інтеграл кореня",
  difficulty: 2,                // рівень; має існувати в `levels` (інакше не покажеться)
  problemTokens: [ ... ],       // умова як токени (див. §4)
  steps: [ ... ],               // кроки рішення (див. §5)
  answer: `...`,                // фінальна відповідь (latex, з кольором)
  methodIds: ["rewrite", "table"], // методи задачі загалом (бейджі під рівнянням)
  pitfalls: [ ... ],            // типові помилки (див. §7)
}
```

Усе інше (картка на дашборді, сторінка, статична генерація, галочка «розв'язано»)
підхоплюється саме по собі. Дашборд показує лише ті рівні, що є в `levels`.

**Новий рівень складності:** додай рядок у `levels` (`difficulty`, `name`, `summary`).

## 4. Як писати ТОКЕНИ рівняння (`Token[]`)

Рівняння — це **масив дрібних шматків**, а не один рядок. Шматок з `hint` стає
інтерактивним (підсвітка + tooltip).

```ts
problemTokens: [
  { latex: "\\int", hint: hintIntegral },            // повторювані — спільні константи
  { latex: "(" },                                    // «клей» без hint
  { latex: `${co("6")}x^{${po("4")}}`, hint: { term: "6x⁴", text: "..." } },
  { latex: "+", hint: hintPlus },
  { latex: `${co("3")}x^{${po("2")}}`, hint: { term: "3x²", text: "..." } },
  { latex: ")" },
  { latex: "dx", hint: hintDx },
]
```

Правила:
- **Розбивай за змістом:** `∫`, кожен доданок, `dx` — окремі токени. На головні
  частини — `hint`. На «клей» (`(`, `)`, `=`, `+`) — переважно без `hint`
  (або спільний, як `hintPlus`).
- **Дужки — окремими токенами `"("` та `")"`**, НЕ `\left(`…`\right)`.
  ⚠️ `\left`/`\right` в РІЗНИХ токенах ламають рендер (KaTeX вимагає пари в одному виразі).
- **Колір** через хелпери з `problems.ts`: `co(...)` — число, `po(...)` — степінь (див. §6).
- Повторювані підказки (`∫`, `dx`, `+`) виноси в константи (`hintIntegral`, `hintDx`, `hintPlus`).

## 5. Як писати КРОКИ (`Step`)

```ts
{
  title: "Виносимо числа наперед",   // дієслово + суть, коротко
  tokens: [ ... ],                   // стан рішення на цей момент (як §4, теж кольорові)
  explanation: "Числа виносимо за знак інтеграла...", // 1–2 речення: що і ЧОМУ
  methodIds: ["linearity"],          // який метод застосовано (бейдж + лінк у модалці)
  help: { ... },                     // модалка «Я не розумію» (див. §8)
}
```

Правила:
- **Гранулярність:** окремий крок на «розбили суму», окремий на «винесли числа»,
  окремий на «застосували формулу», окремий на «спростили». Не поспішай.
- `tokens` кроку теж інтерактивні й кольорові — наведення працює на кожному кроці.
- `methodIds[0]` автоматично дає лінк «Детальніше про метод →» у модалці.
- Останній крок задачі бажано — **перевірка похідною** (`methodIds: ["verify"]`).
- `help` став хоча б на ключові/складні кроки; на тривіальні («збираємо відповідь») можна без нього.

## 6. Кольорове кодування (семантика)

Кольори мають **значення** і тягнуться через УСІ кроки задачі — щоб око
відстежувало елемент.

| Хелпер / колір | Значення |
|---|---|
| `co("...")` → `#fbbf24` бурштин | число (коефіцієнт) |
| `po("...")` → `#34d399` зелений | степінь (показник) |

Хелпери `co`/`po` визначені у `problems.ts` і експортовані як `COLOR_COEF`/`COLOR_POW`.

Приклади консистентності:
- `6x⁴` → `${co("6")}x^{${po("4")}}`; після інтегрування `6·x⁵/5` → у `x⁵` і в `/5`
  той самий `po`, бо це той самий «новий степінь». Студент бачить, куди пішла «5».
- Для інтегрування частинами в теорії — окремі кольори `u/dv/v/du` (див. `theory/page.tsx`).

Правило: **не вводь колір без сенсу.** Якщо елемент нічого не «трасує» — лиши звичайним.

## 7. Як писати ТИПОВІ ПОМИЛКИ (`pitfalls`)

Масив рядків. Можна **жирний** через `**...**` (рендериться компонентом `Pitfalls`).

```ts
pitfalls: [
  "Забути **+ C** — найчастіша втрата балів.",
  "Переплутати з похідною: степінь **росте** (x⁴→x⁵), а не падає.",
]
```

Пиши конкретні граблі **саме цієї** задачі, а не загальні поради.

---

## 8. Як писати ПІДКАЗКИ (`Hint`) і ДОПОМОГУ (`StepHelp`)

### Hint (наведення на токен)
```ts
hint: { term: "6x⁴", text: "Коротко й по суті: що це і що з ним робимо." }
```
- `term` — 1–3 слова (що це).
- `text` — 1 речення простою мовою. Без формул-простирадел.

### StepHelp (модалка «Я не розумію») — багаторівнева

```ts
help: {
  plain: "Пояснення на пальцях (рівень 1).",
  simpler: "ЩЕ простіше, якщо перше не зайшло (рівень 2). Аналогія/побутово.",
  numeric: [                       // те саме на КОНКРЕТНИХ числах, без іксів
    { latex: "\\int x^{4}\\,dx", note: "показник = 4" },
    { latex: "4 + 1 = 5", note: "рух 1: +1" },
    { latex: "\\frac{x^{5}}{5}", note: "рух 2: ділимо на 5" },
  ],
  examples: [                      // ще приклади того ж правила
    { latex: "\\int x^2\\,dx = \\frac{x^3}{3}", note: "2→3, ділимо на 3" },
  ],
  check: {                         // міні-самоперевірка
    question: "Чому дорівнює ∫x⁵ dx?",
    answer: "\\frac{x^{6}}{6} + C",
  },
}
```

Правила:
- `plain` — обов'язково. Решта — опційно, модалка рендерить лише наявне.
- На **складних/ключових** кроках додавай `simpler` + `numeric` + `check` (повний набір).
- `numeric` має вести **по кроках на реальних числах** — це рятує тих, хто не дружить з `x`.
- `check.answer` — у latex; розкривається кнопкою (активне пригадування).
- Лінк на метод НЕ дублюй у `help` — він береться зі `step.methodIds[0]`.

---

## 9. KaTeX: технічні правила

- У TS-рядках екрануй бекслеш подвійним: `"\\int"`, `"\\frac{a}{b}"`, `"\\,"`.
- `\left(`…`\right)` — **тільки в межах одного** latex-рядка/токена, не розривай по токенах.
- Колір — `\\textcolor{#hex}{...}` (хелпери `co`/`po` уже це роблять).
- Рендер стійкий до помилок (`throwOnError:false`), але **перевір візуально** — мовчазний
  фолбек покаже сирий latex.

## 10. Чекліст перед комітом

1. `npm run build` — без помилок, усі сторінки генеруються.
2. `npm run lint` — чисто.
3. У браузері (`npm run dev`):
   - формула умови й кожного кроку рендериться правильно (дужки, дроби, степені);
   - наведення дає підказки скрізь (умова + кожен крок);
   - модалка «Я не розумію»: `ще простіше`, `на числах`, `самоперевірка`, лінк на метод;
   - бейджі методів стоять і відкривають опис;
   - кольори консистентні (одна «5» зелена скрізь, де це той самий степінь).
4. **Математика:** візьми похідну від `answer` → має вийти підінтегральна функція.
5. Тон і простота: чи зрозуміє це першокурсник без сторонньої допомоги?

## 11. Антипатерни (так НЕ робимо)

- ❌ Рівняння одним суцільним `latex`-рядком без токенів (втрачаємо наведення).
- ❌ `\left(` і `\right)` у різних токенах (зламаний рендер).
- ❌ Крок, що робить одразу 2–3 перетворення.
- ❌ Колір «для краси» без семантики.
- ❌ Підказка з ще складнішою мовою, ніж сама формула.
- ❌ Новий метод, що дублює наявний (степеневе правило окремо від `table`).
- ❌ Відповідь без `+ C`.
- ❌ `help.plain` як єдиний рівень на складному кроці (дай `simpler`/`numeric`).
